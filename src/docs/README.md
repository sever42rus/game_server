# Документация серверной части

`src` содержит TCP-сервер игрового мира. Сервер принимает бинарные команды от
клиентов, обновляет доменное состояние игроков и рассылает клиентам бинарные
пакеты состояния мира.

## Запуск

```bash
python -m src.main
```

По умолчанию сервер слушает `127.0.0.1:9000`. Настройки находятся в
`src/infrastructure/config.py`:

- `HOST` - адрес TCP-сервера;
- `PORT` - порт TCP-сервера;
- `TICK_RATE` - частота игрового цикла;
- `STATE_CHANGE_COOLDOWN_SECONDS` - длительность блокировки смены состояния
  после переходов `standing <-> sitting`;
- `COMMAND_QUEUE_SIZE` - максимальный размер очереди входящих команд.

## Структура

- `src/domain` - доменная модель игры: игроки, позиции, состояния персонажа,
  команды и сервис перемещения.
- `src/application` - сценарии приложения, DTO между доменом и
  инфраструктурой, сборка снимков мира для клиентского протокола.
- `src/infrastructure` - TCP-сервер, бинарный кодек протокола, реестр сессий
  и отправка сообщений клиентам.
- `src/main.py` - композиционный корень, который собирает зависимости и
  запускает сервер.

## Игровой цикл

1. Клиент подключается по TCP.
2. Сервер создает или находит `player_id` для соединения и ждет пакет `SET_NAME`.
3. Клиент отправляет обязательное имя игрока в `UTF-8`.
4. Только после успешной валидации имени сервер добавляет игрока в `WorldState`.
5. Клиент получает `OWN_PLAYER`, затем серию `PLAYER_SPAWN` и пакет `SNAPSHOT_END`.
6. Сервер рассылает всем остальным клиентам `PLAYER_SPAWNED`.
7. Входящие игровые команды складываются в очередь.
8. Фоновый цикл с частотой `TICK_RATE` обрабатывает команды, двигает игроков и
   рассылает события.

Движение моделируется на сервере. При старте движения сервер отправляет
`PLAYER_MOVE_STARTED`, а при достижении цели или смене состояния отправляет
`PLAYER_UPDATED`.

## Входящие сообщения клиента

Входящие команды передаются в бинарном little-endian формате. Первые 2 байта
каждого пакета содержат `opcode`, затем идут поля payload для конкретной
команды.

Пакет установки имени `SET_NAME` (`0x0100`):

```text
[opcode:uint16][name_length:uint8][name:utf8 bytes]
```

Это должен быть первый клиентский пакет после открытия TCP-соединения.
Сервер принимает только непустые имена длиной до `24` байт в `UTF-8` после
нормализации пробелов по краям.

Команда движения `MOVE` (`0x0101`):

```text
[opcode:uint16][target_x:int32][target_y:int32][movement:uint8]
```

Код режима движения:

- `1` - ходьба;
- `2` - бег.

Если у игрока была выбрана цель, успешный старт движения автоматически
сбрасывает ее. В этом случае сервер сначала отправляет `PLAYER_TARGET_CHANGED`
без выбранной цели, затем `PLAYER_MOVE_STARTED`.

Команда `STAND` (`0x0102`) не содержит payload и переводит персонажа в
состояние `standing`.

Команда `SIT` (`0x0103`) не содержит payload и переводит персонажа в состояние
`sitting`.

Команда состояния останавливает текущее движение игрока и переводит персонажа
в указанное состояние. Если переход между состояниями запрещен доменными
правилами, команда игнорируется.

После перехода `standing -> sitting` или `sitting -> standing` персонаж не
может менять свое состояние в течение времени, заданного настройкой
`STATE_CHANGE_COOLDOWN_SECONDS`. В этот период также игнорируются команды
начала движения, так как они тоже меняют состояние персонажа.

Команда выбора цели `SELECT_TARGET` (`0x0104`):

```text
[opcode:uint16][target_type:uint8][target_id:uuid(16 bytes)]
```

Код типа цели:

- `1` - игрок.

Выбор несуществующего игрока игнорируется. Сброс выбранной цели выполняется
пакетом `CLEAR_TARGET` (`0x0105`) без payload.

Если выбранный игрок отключается, сервер очищает ссылки на него в состоянии
мира. Клиент также получает пакет `PLAYER_DISCONNECTED` для локальной очистки.

## Исходящие сообщения сервера

Начальная загрузка мира новому клиенту состоит из серии бинарных пакетов:

1. `OWN_PLAYER` — передает `UUID` локального игрока.
2. `PLAYER_SPAWN` — создает одного уже видимого игрока в локальном мире клиента.
3. `SNAPSHOT_END` — завершает начальную инициализацию локального мира.

После этого сервер отправляет live-пакеты:

- `PLAYER_SPAWNED` — появился новый игрок;
- `PLAYER_MOVE_STARTED` — игрок начал движение к точке;
- `PLAYER_UPDATED` — игрок синхронизирован в точной точке и состоянии;
- `PLAYER_TARGET_CHANGED` — игрок выбрал или сбросил цель;
- `PLAYER_DISCONNECTED` — игрок удален из локального мира клиента.

Пакеты `PLAYER_SPAWN` и `PLAYER_SPAWNED` дополнительно передают имя игрока как
часть профиля персонажа.

В пакете `PLAYER_SPAWN` могут дополнительно присутствовать:

- выбранная цель игрока;
- целевая точка текущего движения, если игрок уже двигался в момент входа клиента.

Если клиент запускает движение с кодом `movement = 1`, сервер передает
`state = 2` и скорость `50`. Для `movement = 2` сервер передает `state = 3`
и скорость `100`.

## Состояния персонажа

- `standing` - персонаж стоит;
- `walking` - персонаж идет к целевой точке со скоростью `50%`;
- `running` - персонаж движется к целевой точке;
- `sitting` - персонаж сидит.

## Переходы между состояниями

- из `standing` можно перейти в `standing`, `walking`, `running`, `sitting`;
- из `walking` можно перейти в `standing`, `walking`, `running`;
- из `running` можно перейти в `standing`, `walking`, `running`;
- из `sitting` можно перейти только в `standing`.

Команды, нарушающие эти правила, игнорируются. Из состояния `sitting`
персонаж не может начать движение, пока не получит команду `stand`.

После переходов `standing -> sitting` и `sitting -> standing` включается
блокировка смены состояния на время из `STATE_CHANGE_COOLDOWN_SECONDS`
игровых секунд. Пока блокировка активна, персонаж не может перейти в
`walking`, `running`, `standing` или `sitting`, если это требует смены
текущего состояния.

## Архитектурные правила

- Доменный слой не должен зависеть от TCP, бинарного протокола или asyncio.
- Доменный слой хранит состояние мира, но не собирает клиентские снимки мира.
- `GameService` собирает `WorldSnapshotDTO`, чтобы новые типы сущностей мира
  добавлялись на уровне сценариев приложения.
- Валидация транспортного формата находится в `GameProtocolCodec`.
- `GameService` координирует доменную модель, но не знает о TCP-соединениях.
- Новые публичные и приватные функции должны иметь докстринги на русском языке.
